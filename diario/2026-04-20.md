### 3. Decisão Arquitetural: Irrigação no Balanço
Pesquisa dos 10 top sistemas (Valley, FieldNET, Netafim, CropX, Manna, Hortau, WaterBit, Arable):
- **Nenhum** assume irrigação de schedule como executada
- Todos exigem confirmação (telemetria, sensor de solo, ou manual)
- Implementado padrão "Schedule + Confirm Exceptions": cron auto-confirma planned→done
- Revertida contabilização de irrigação programada na projeção (FAO-56: Ii = lâmina real)

## Decisões técnicas

| Decisão | Motivo |
|---|---|
| Gap-fill sem API externa | Chamadas Open-Meteo ao vivo são não-determinísticas; agricultor via valores diferentes a cada reload |
| Irrigação só com confirmação real | FAO-56 Eq. 85: Ii deve ser lâmina realmente aplicada. Nenhum sistema top assume schedule=executado |
| Schedule + Confirm Exceptions | Cron auto-confirma planned→done. Agricultor só age na exceção. Escala para 20+ pivôs |
| FC% = adcNew/cta (sem peak) | peakReachedCta causava gauge 100% quando real era 95% — contradição com gráfico |
| Projeção usa pivot > season > default | Mesma cadeia de fallback para CC/PM/Ds em todos os pontos (dashboard, manejo, cron) |
| HistoryBlock = pivô mais crítico | Steve Jobs: "se o usuário precisa escolher, você não sabe o que mostrar" |

- Dados fake (efficiency=94, MoistureChart hardcoded)
